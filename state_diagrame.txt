0) High-Level State Flow (Text)
================================================================================

Project_Boot
  -> Config_SSOT (config.json)
  -> Engine_Core (signal + plan + metrics + blocked_by)
  -> API_Server (signal_preview + ai_confirm)
  -> Mentor_Executor (orchestrator)
      -> [Phase 7] Synchronous KILL_SWITCH.txt check (fail-closed)
          -> If KILL_SWITCH.txt EXISTS => SKIP immediately (status=SKIP, reason=kill_switch_active)
          -> If KILL_SWITCH.txt ABSENT => Continue to AI confirm
  -> AI_Final_Confirm (confirm-only)
  -> MT5_Executor_Hardening (fail-closed)
  -> MT5_OrderSend
  -> Logs/Stats/Notify

Kill-Switch Defense in Depth (Phase 7)
  Layer 1 (Sync): mentor_executor.py checks KILL_SWITCH.txt before EVERY execution
    - Zero trades after breach (eliminates 1–6 second enforcement gap)
  Layer 2 (Async): watchdog_supervisor.py forces DRY_RUN=1 on mentor restart
    - Async backup: ensures DRY_RUN persists across restarts
  Layer 3 (Write): risk_guard_hardstop.py writes KILL_SWITCH.txt on breach
    - Source of truth: writes the file that Layers 1 & 2 read

Deterministic Path (Phase 7)
  All three components resolve KILL_SWITCH.txt using:
    PROJECT_ROOT = os.path.dirname(os.path.abspath(__file__))
    KILL_SWITCH_PATH = os.path.join(PROJECT_ROOT, "KILL_SWITCH.txt")
  This ensures the same file is used regardless of the working directory
  at process start time.

Core invariants (ห้ามเปลี่ยนหลักการ)
- Confirm-only: AI อนุญาต/ปฏิเสธเท่านั้น และต้องไม่แก้ plan (entry/sl/tp)
- Fail-closed: ข้อมูลขาด/ผิดรูปแบบ/เชื่อมต่อไม่ได้ => SKIP ไม่เทรด
- Fail-closed (Phase 7): KILL_SWITCH.txt exists or read error => SKIP ไม่เทรด
- Telegram separation:
  - Trade room: TELEGRAM_TRADE_CHAT_ID (trade/execution alerts)
  - Mentor room: TELEGRAM_MENTOR_CHAT_ID (read-only/teaching/diagnostics)
  - Guard: ถ้า mentor_chat_id == trade_chat_id จะ block การส่งเพื่อกันปนห้อง

- Logs event: kill_switch_block with request_id and kill_switch_reason

================================================================================
H) KILL_SWITCH.txt — Fail-Safe Protocol
================================================================================

File location: {script_dir}/KILL_SWITCH.txt
  where {script_dir} = directory containing each Python file (all in project root)

Who WRITES it:
  - risk_guard_hardstop.py: writes on daily DD / total DD / open positions breach
  - Operator: manual write for emergency stop

Who READS it:
  - mentor_executor.py (Phase 7): synchronous check before every execution
  - watchdog_supervisor.py: async check; forces DRY_RUN=1 on mentor restart

Behavior matrix:
  | File State       | mentor_executor response        | watchdog response           |
  |------------------|---------------------------------|-----------------------------|
  | File absent      | Continue (execute/AI/DRY_RUN)   | Normal restart              |
  | File exists, OK  | SKIP (kill_switch_active)       | Restart with DRY_RUN=1      |
  | File exists, ERR | SKIP (KILL_SWITCH_READ_ERROR)   | Restart with DRY_RUN=1      |

File format: Plain text, first 500 chars used as kill_switch_reason
  Example: "DAILY_DD_BREACH: -5.2% at 2026-03-13T05:00:00"

To resume trading:
  1. Investigate breach
  2. Confirm breach is resolved
  3. Delete KILL_SWITCH.txt
  4. Restart mentor_executor (watchdog will restart without DRY_RUN=1)
